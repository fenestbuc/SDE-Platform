# SDE-Platform V4 Implementation Plan

## Goal
Evolve the stabilized SDE-Platform (V3) into a production-ready, highly secure, and scalable messaging platform. This next version will focus on advanced cryptographic properties (Forward Secrecy, Authenticity), infrastructure scaling, and a comprehensive UX overhaul.

## Current Context & Assumptions
- **Base Architecture:** The V3 rewrite is complete and stable (Express, Postgres, Prisma, React/Vite).
- **Testing:** Unit (Vitest) and E2E (Playwright) test suites are fully green, including complex encrypted chunked file uploads and WebSocket real-time delivery.
- **Current Crypto:** Uses static ECIES (secp256k1) + AES-GCM. It encrypts payloads correctly but lacks Perfect Forward Secrecy (PFS) and explicit digital signatures.
- **Current Real-time:** WebSockets are implemented in-memory, which prevents the Node.js backend from scaling horizontally to multiple instances.

## Proposed Approach
We will divide the V4 development into three distinct phases to manage risk:
1. **Cryptography & Security:** Introduce digital signatures to prove message authenticity and lay the groundwork for Perfect Forward Secrecy (PFS).
2. **Infrastructure & Scalability:** Replace the in-memory WebSocket registry with a Redis-backed Pub/Sub architecture to enable horizontal scaling.
3. **UX & Feature Polish:** Overhaul the raw HTML/Tailwind frontend using a component library (e.g., shadcn/ui) and finalize the Web Push Notification pipeline.

## Step-by-Step Plan

### Phase 1: Advanced Cryptography (Authenticity & Security)
- **Message Signing:** 
  - Update `src/crypto/ecies.ts` to require the sender to sign the ciphertext (or a hash of the ciphertext + ephemeral public key) using ECDSA.
  - Update `Message` schema in `prisma/schema.prisma` to store the `signature`.
  - Update the React crypto worker to verify the signature against the sender's known public key before decrypting.
- **Forward Secrecy Foundation:**
  - Introduce an initial key-rotation mechanism or a simplified Double Ratchet step so a compromised long-term private key doesn't decrypt historical messages.

### Phase 2: Horizontal Scalability
- **Redis Integration:**
  - Add Redis to `docker-compose.yml`.
  - Introduce `ioredis` or `redis` npm package to the backend.
- **WebSocket Pub/Sub:**
  - Refactor `src/websocket/handler.ts` and `src/websocket/server.ts`.
  - When a message is sent to user X, publish it to a Redis channel `user:X`.
  - WebSocket servers subscribe to Redis channels for their connected clients, allowing messages to span across multiple Node.js processes.

### Phase 3: UX Overhaul & Push Notifications
- **Frontend Refactor:**
  - Migrate `Dashboard.tsx` to a modular layout with proper React components (Sidebar, MessageList, ComposeModal).
  - Add loading skeletons, error boundaries, and toast notifications.
- **Web Push Notifications:**
  - The frontend already subscribes to push in `App.tsx`.
  - Implement the backend logic using the `web-push` library to actually trigger notifications when an offline user receives a message.

## Tests / Validation
- **Crypto Unit Tests:** Must verify that invalid ECDSA signatures correctly reject decryption attempts.
- **E2E Playwright:** Existing flows (Registration, Messaging, File Upload) must continue to pass seamlessly with the new signing logic and Redis pub/sub layer.
- **Scale Test:** Boot up 2 Node.js instances behind a load balancer and verify WebSocket messages are routed correctly via Redis.

## Risks, Tradeoffs, and Open Questions
- **Risk:** Implementing Perfect Forward Secrecy (PFS) is notoriously difficult to get right. We will start with strict Message Signing (easier) before full PFS.
- **Tradeoff:** Adding Redis increases operational complexity. For a small deployment, in-memory is fine, but for "V4 Scale", Redis is a hard requirement.
- **Open Question:** Do we want to support multi-device sync? If so, the key management model becomes significantly more complex (requiring Sender Keys or syncing private keys securely between devices).
