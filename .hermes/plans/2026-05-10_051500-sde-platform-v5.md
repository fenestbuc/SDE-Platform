# SDE-Platform v5: Infinite Scale File Transfers & Offline Push Notifications

**Project:** fenestbuc/SDE-Platform  
**Current State:** v4 (Completed) - Chunked client-side encryption, Web Worker isolation, Playwright E2E tests.  
**Goal:** Resolve the final architectural bottleneck (backend memory limits for file uploads) and introduce offline capabilities for a true enterprise-grade E2EE messenger experience.

---

## 1. Current Context & Limitations

While v4 successfully implemented **client-side** chunking to prevent browser OOM (Out-of-Memory) crashes, the **backend** still relies on `multer` which buffers file uploads through the Express server and enforces a hard 50MB limit (`limits: { fileSize: 50 * 1024 * 1024 }`). 

Uploading a 1GB file through an Express Node.js server introduces massive latency, CPU load, and bandwidth costs. Furthermore, users currently only receive notifications if they have the WebSocket actively open in their browser tab.

## 2. Proposed Approach for v5

To achieve "Infinite Scale" file transfers and better UX, we will implement:

1. **Direct-to-S3 Presigned Multipart Uploads:** 
   - The Express backend will no longer process file bytes. Instead, it will generate AWS S3 Presigned URLs.
   - The React frontend will encrypt chunks via the Web Worker and immediately stream them directly to the S3 bucket (or Tigris/Cloudflare R2).
   - This effectively removes all file size limits on the Node.js API, allowing terabyte-scale E2EE transfers.

2. **Web Push API (Service Workers):**
   - Introduce a Service Worker to the React frontend to listen for Web Push notifications.
   - Integrate `web-push` on the Node.js backend. When a message is sent to an offline user, the backend will trigger a push notification ("New encrypted message from Alice") without revealing the plaintext.

## 3. Step-by-Step Plan

### Phase 1: Backend Direct-to-S3 Architecture
1. Add `@aws-sdk/s3-request-presigner` to the backend.
2. Create a new API endpoint `POST /api/messages/upload-url` that generates pre-signed PUT URLs for S3.
3. For large files, implement S3 Multipart Upload endpoints (`CreateMultipartUpload`, `UploadPart`, `CompleteMultipartUpload`).
4. Modify `MessageService` so `sendMessage` accepts a `storagePath` (S3 object key) instead of a raw `Multer.File` buffer.

### Phase 2: Frontend Direct Upload Integration
1. Update `Dashboard.tsx` to handle the new upload flow:
   - Request a presigned URL (or multipart URLs) from the backend.
   - As the Web Worker encrypts each 5MB chunk, immediately `fetch(PUT)` that chunk directly to S3.
2. Remove the bloated `FormData` file attachment logic from the main `POST /api/messages` request.

### Phase 3: Web Push Notifications
1. Generate VAPID keys for the backend.
2. Add a `PushSubscription` model to the Prisma schema (1 User -> Many Subscriptions).
3. Create an endpoint `POST /api/notifications/subscribe` to save client push subscriptions.
4. Implement a Service Worker (`public/sw.js`) in the React app to receive and display system-level notifications.
5. Update `MessageService.sendMessage` to trigger a Web Push notification if the recipient is not currently connected via WebSocket.

### Phase 4: Testing & Cleanup
1. Remove `multer` dependency as the backend no longer handles binary file streams.
2. Update Playwright E2E tests (`tests/e2e/messaging.spec.ts`) to mock the S3 presigned URL endpoints.
3. Test service worker registration in Playwright.

## 5. Risks, Tradeoffs, and Open Questions

- **Risk:** Presigned URLs bypass the backend's rate limiters for the actual file upload bandwidth.
  - *Mitigation:* We will generate URLs with strict expirations (e.g., 15 minutes) and enforce content-length restrictions in the S3 bucket policies.
- **Tradeoff:** Local development without an S3 bucket becomes slightly harder.
  - *Mitigation:* We can spin up a local `MinIO` instance in `docker-compose.yml` to act as the S3 target for local testing.
- **Open Question:** Should we implement full Signal Protocol (Double Ratchet) for Perfect Forward Secrecy (PFS) in this version? 
  - *Recommendation:* Defer to v6. Removing the Node.js file bottleneck is more critical for platform stability right now.
