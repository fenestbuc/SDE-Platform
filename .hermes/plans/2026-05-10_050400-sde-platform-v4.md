# SDE-Platform v4: Enterprise Scalability & Quality Assurance Plan

**Project:** fenestbuc/SDE-Platform
**Current State:** v3 (Completed) - React frontend, Web Worker cryptography isolation, S3 storage abstraction, and 0 `npm audit` vulnerabilities.
**Goal:** Address the final limitations identified in the v3 audit: replace placeholder E2E tests with comprehensive Playwright flows, and implement chunked file encryption to support massive file uploads without causing browser Out-Of-Memory (OOM) crashes.

---

## 1. Current Context & Limitations

The v3 audit highlighted two primary areas for improvement:
1.  **Testing Gap:** The E2E test file (`tests/e2e/flow.spec.ts`) contains a placeholder. We need real browser automation to ensure the cryptographic worker, React UI, and Express API function together correctly in a production-like environment.
2.  **Large File Bottleneck:** The Web Worker currently reads the entire file into a single `ArrayBuffer` before passing it to `crypto.subtle.encrypt()`. Browsers strictly limit contiguous memory allocations, meaning users cannot securely upload very large files (e.g., 1GB+) without crashing the tab.

## 2. Proposed Approach

**For E2E Testing:**
Write full Playwright user journeys that interact with the React DOM to register two separate users, have User A send an encrypted text and file message to User B, and have User B login to successfully decrypt and download them.

**For Chunked Streaming Encryption:**
Transition from monolithic `ArrayBuffer` encryption to a chunked strategy using the Web Streams API (`ReadableStream` and `WritableStream`):
-   Slice the file into manageable blocks (e.g., 5MB).
-   Encrypt each block individually with AES-GCM. 
-   To maintain cryptographic integrity, derive a base IV from HKDF and increment it deterministically for each chunk (or generate random IVs per chunk and prepend them).
-   Assemble the encrypted chunks into a browser `Blob` (which is disk-backed, saving RAM) before sending it via `FormData`, or implement multipart/chunked uploading directly to the backend.

---

## 3. Step-by-Step Plan

### Phase 1: Comprehensive E2E Testing
1.  Setup Playwright to run the local Vite dev server and backend simultaneously during `test:e2e`.
2.  Write `auth.spec.ts`: Test user registration, login, JWT rotation, and logout.
3.  Write `messaging.spec.ts`: Test the full ECIES exchange. User A logs in, fetches User B's public key, encrypts a message via the Web Worker, and sends it. User B logs in, receives it, and decrypts it.
4.  Ensure CI (`.github/workflows/ci.yml`) is configured to run Playwright headless browsers.

### Phase 2: Web Worker Streaming Crypto
1.  Modify `src/client/src/workers/crypto.worker.ts` to accept `File` objects directly (passed via `postMessage` which supports File transfer without reading to RAM).
2.  Implement a chunking loop inside the worker:
    *   Read a 5MB slice.
    *   Encrypt the slice with AES-GCM (generating a new IV per chunk, prepended to the ciphertext).
    *   Append the encrypted chunk to an array of Blobs.
3.  Return a single concatenated `Blob` back to the main thread. This offloads the storage to the browser's disk-backed Blob storage rather than RAM.
4.  Implement the inverse for decryption: read Blob slices, extract IV, decrypt, and construct a plaintext Blob.

### Phase 3: Backend Multipart Uploads (Optional but Recommended)
*If standard FormData Blob uploading is still too heavy for network transmission:*
1.  Refactor `routes/messages.ts` and `fileService.ts` to accept chunked file uploads (e.g., via `multer` streaming or direct S3 multipart uploads).
2.  Update the frontend `Dashboard.tsx` to upload encrypted chunks sequentially to the backend rather than waiting for the entire file to be encrypted locally.

### Phase 4: UI/UX Refinement
1.  Update the Compose UI to display a granular progress bar for both the encryption phase (Worker progress) and the network upload phase.
2.  Gracefully handle cancellation of large file encryptions/uploads.

## 5. Risks, Tradeoffs, and Open Questions

-   **Risk:** `crypto.subtle.encrypt` using AES-GCM is not natively designed for continuous streaming of a single continuous ciphertext. By chunking, we are creating multiple independent AES-GCM ciphertexts. 
-   **Tradeoff:** Each chunk will have its own IV (12 bytes) and Auth Tag (16 bytes). We must store these within the file blob (e.g., `[IV][Ciphertext][Tag]`) so the decrypter knows how to parse it. This adds slight file size overhead (28 bytes per chunk), which is negligible.
-   **Open Question:** Do we want to build true multipart streaming to the backend (Phase 3), or just fix the RAM issue via Blobs (Phase 2) and upload the whole 1GB encrypted Blob in one request? *Recommendation:* Start with Phase 2 (Blob disk-backing) as it solves the immediate OOM crash. Phase 3 is a larger infrastructural change.
