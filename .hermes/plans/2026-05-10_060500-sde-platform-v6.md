# SDE-Platform v6: Perfect Forward Secrecy, Group Messaging, and Test Stabilization

**Project:** fenestbuc/SDE-Platform  
**Current State:** v5 (Completed) - Zero-knowledge architecture, Direct-to-S3 streaming, Web Worker cryptography, and Web Push.  
**Goal:** Elevate the cryptographic model from static ECIES to state-of-the-art Perfect Forward Secrecy (PFS), introduce End-to-End Encrypted Group Messaging, and resolve the flaky Playwright test environment bindings.

---

## 1. Current Context & Limitations

The v5 architecture is highly secure and scalable, but it has a few conceptual limitations compared to modern enterprise messengers (like Signal or WhatsApp):
1. **Static Public Keys:** Currently, a user's static public key is used for all incoming messages. If a user's private key is somehow compromised in the future, all past messages encrypted with that key could theoretically be decrypted (no Perfect Forward Secrecy).
2. **1-to-1 Only:** The platform currently only supports direct messaging. True enterprise platforms require group communications.
3. **Flaky Test State:** Playwright E2E tests intermittently fail because the `webServer` block boots the server against the same SQLite database (`dev.db`) used for development, causing Prisma lock contention and state bleeding between test runs.

## 2. Proposed Approach for v6

We will implement the following major upgrades:

1. **Double Ratchet / PFS (Perfect Forward Secrecy):** 
   - Migrate from static ECIES to a session-based ratchet (inspired by the Signal Protocol). 
   - Users will publish a bundle of "PreKeys" to the server.
   - Senders fetch a PreKey to initiate a session, ensuring new ephemeral keys are mixed in continually.
2. **E2EE Group Messaging:**
   - Implement "Sender Keys" for groups. Instead of encrypting a message N times for N participants, the sender generates a Sender Key, encrypts it individually for each participant (using 1-to-1 sessions), and then encrypts the actual message payload symmetrically once using the Sender Key.
3. **Test Environment Isolation:**
   - Implement a dedicated test database (`test.db`) and configure Playwright to inject `NODE_ENV=test`. Add global setup/teardown hooks to reset the database cleanly before every test run.

## 3. Step-by-Step Plan

### Phase 1: Test Environment Stabilization
1. Create `prisma/schema.test.prisma` or use `dotenv` overrides to force Prisma to use `file:./test.db` during `npm run test:e2e`.
2. Create a `tests/e2e/global.setup.ts` script to run `prisma migrate reset --force` before Playwright boots.
3. Update `playwright.config.ts` to utilize the new isolated environment, preventing port collisions and database locks.

### Phase 2: Perfect Forward Secrecy (X3DH & Double Ratchet)
1. **Database Schema:** 
   - Add a `PreKey` model representing one-time use keys.
   - Add a `SessionState` model (encrypted) for users to sync their ratchet state across devices.
2. **Backend API:** 
   - `POST /api/keys/prekeys` (Upload PreKey bundles).
   - `GET /api/keys/bundle/:userId` (Fetch a recipient's PreKey bundle to start a session).
3. **Web Worker (`crypto.worker.ts`):**
   - Implement X3DH (Extended Triple Diffie-Hellman) for initial key agreement.
   - Implement symmetric KDF chains (Double Ratchet) for deriving unique message keys for every single payload.

### Phase 3: Group Encrypted Messaging
1. **Database Schema:**
   - Add `Group` and `GroupMember` models.
   - Update `Message` to optionally point to a `groupId` instead of a `recipientId`.
2. **Backend API:**
   - `POST /api/groups` (Create group, add members).
   - `POST /api/groups/:id/messages` (Send a group message).
3. **Frontend / Web Worker:**
   - When a group is created, the creator's worker generates a symmetric `SenderKey`.
   - The worker wraps this `SenderKey` in the 1-to-1 PFS sessions of all members.
   - Subsequent group messages are encrypted with AES-GCM using the `SenderKey`.

### Phase 4: UI/UX Integration
1. Update `Dashboard.tsx` to include a "Groups" tab.
2. Add a "Create Group" modal with user multi-select.
3. Update the message rendering logic to handle group contexts (displaying sender names inline).

## 5. Risks, Tradeoffs, and Open Questions

- **Risk:** Implementing the Double Ratchet protocol from scratch in JavaScript is cryptographically complex and prone to subtle vulnerabilities.
  - *Mitigation:* We will rely on established open-source libraries like `libsignal-protocol-javascript` if possible, or build a strictly scoped, simplified ratchet utilizing `@noble/curves` following the exact Signal specification.
- **Tradeoff:** Perfect Forward Secrecy requires maintaining state (the ratchet/chain keys). If a user clears their browser cache, they lose their session state and cannot decrypt new messages until a new session is established.
  - *Mitigation:* We will store encrypted session states on the server, allowing the client to pull down and decrypt their ratchet state using their master password key.
- **Open Question:** Do we want to support multi-device syncing for the same user?
  - *Recommendation:* Keep it single-device per user for v6. Multi-device E2EE requires distributing session keys across a user's own device graph, which massively inflates the scope.
