# SDE-Platform v3: Scalability, Security Hardening, and Modernization Plan

**Project:** fenestbuc/SDE-Platform
**Current State:** v2 (Completed) - Functional E2EE messaging platform with Vanilla JS frontend, Express backend, SQLite, and WebSockets.
**Goal:** Address the technical debt and limitations identified during the v2 audit, transitioning the application from a robust portfolio demo into a highly scalable, enterprise-ready system.

---

## 1. Current Context & v2 Findings

The v2 audit revealed several architectural bottlenecks that need to be resolved for true production readiness:
1.  **File Size Limits (Memory OOM):** Web Crypto's `AES-GCM` requires the entire file to be loaded into memory as an `ArrayBuffer`. This restricts file uploads to ~50MB.
2.  **State Management & UI Complexity:** As the application grows, managing WebSocket state, encryption keys, and UI reactivity in Vanilla JS is becoming unwieldy.
3.  **Key Exposure Risk (XSS):** The decrypted private key currently resides in `sessionStorage`. While mitigated by CSP, an XSS vulnerability could allow a malicious script to exfiltrate the plaintext private key.
4.  **Local Storage Bottleneck:** Encrypted file blobs are currently saved to the local disk (`/uploads`). This breaks horizontal scaling and is ephemeral on platforms like Fly.io unless volumes are perfectly managed.
5.  **Security Audit Warnings:** `npm audit` flagged high-severity vulnerabilities in `tar` (a sub-dependency of `bcrypt` via `node-pre-gyp`).

## 2. Proposed Approach for v3

We will iterate on the existing foundation with the following major upgrades:
-   **Chunked Streaming Encryption:** Implement a chunked encryption/decryption pipeline (e.g., streaming chunks of 5MB) using Web Streams API to support arbitrary file sizes (up to 1GB+).
-   **S3-Compatible Object Storage:** Refactor `FileService` to upload encrypted file blobs directly to an S3-compatible store (AWS S3, Cloudflare R2, or Tigris).
-   **Frontend Framework Migration:** Migrate the frontend to **React (via Vite)** to better handle complex state (WebSockets, chunking progress, key management) with clean components.
-   **Web Worker Key Vault:** Move cryptographic operations and private key storage into a dedicated **Web Worker**. The main thread will never have access to the raw private key, neutralizing the XSS risk.
-   **Dependency Hardening:** Replace `bcrypt` with `bcryptjs` (pure JS) or `argon2` to eliminate the `node-pre-gyp` vulnerability chain.

## 3. Step-by-Step Plan

### Phase 1: Backend Storage & Security Hardening
1.  Swap `bcrypt` for `bcryptjs` or `argon2` to clear `npm audit` vulnerabilities.
2.  Refactor `src/services/fileService.ts` to implement the AWS SDK (`@aws-sdk/client-s3`).
3.  Add environment variables for S3 (`S3_BUCKET`, `S3_REGION`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`).
4.  Update API endpoints to support multi-part chunked uploads or pre-signed URLs (to bypass the Node server entirely and upload directly to S3).

### Phase 2: Frontend Migration (React Setup)
1.  Initialize a new Vite + React + TypeScript project in a `client/` or `web/` directory.
2.  Port the existing Tailwind CSS styles.
3.  Re-implement the routing (Login, Register, Dashboard, Admin) using `react-router-dom`.
4.  Implement a centralized store (e.g., Zustand or React Context) for WebSocket connections and authentication state.

### Phase 3: Cryptography Web Worker & Chunking
1.  Create a Web Worker (`crypto.worker.ts`) responsible for holding the user's decrypted private key in its isolated memory scope.
2.  Expose message-passing interfaces so the React app can request the Worker to encrypt/decrypt payloads without ever touching the key.
3.  Implement Streams API (`ReadableStream` / `WritableStream`) in the Worker to encrypt and decrypt file chunks sequentially.
4.  Integrate chunked uploading with the backend/S3.

### Phase 4: CI/CD & Production Rollout
1.  Update the Dockerfile to build both the Express API and the React static assets.
2.  Configure Express to serve the built React app from `client/dist`.
3.  Update Playwright E2E tests to interact with the new React DOM structure.
4.  Deploy to Fly.io, configuring the necessary S3 secrets and Tigris object storage bucket.

## 5. Tests / Validation

-   **Unit Tests:** Verify that `bcryptjs`/`argon2` correctly validates existing hashed passwords.
-   **Integration Tests:** Mock the AWS S3 client and ensure `fileService` correctly puts and gets objects.
-   **E2E Tests:** Use Playwright to upload a large dummy file (e.g., 200MB) to verify the chunking logic does not crash the browser tab.
-   **Security Validation:** Manually attempt to access the private key from the browser DevTools console to ensure the Web Worker isolation holds.

---

## 6. Risks, Tradeoffs, and Open Questions

-   **Risk:** Migrating to React is a significant architectural shift. *Mitigation:* We will keep the API interface identical where possible to isolate the frontend rewrite from backend logic.
-   **Tradeoff:** Chunked encryption is more complex than standard AES-GCM. We must ensure that the AES-GCM authentication tags are validated per chunk, or switch to a stream-native cipher if supported by Web Crypto (or polyfill ChaCha20-Poly1305).
-   **Open Question:** Should we implement pre-signed S3 URLs for client-direct uploads, or pipe the chunks through the Express server? *Recommendation:* Client-direct S3 uploads save server bandwidth but complicate the auth/crypto model. Piping through Express is easier to implement first.
